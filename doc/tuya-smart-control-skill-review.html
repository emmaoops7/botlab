一、Frontmatter / Description — 触发效率低

当前问题

description 字段充斥营销语言，而非帮助 AI Agent 理解"何时触发"的功能性描述：

"industry's broadest AI + device interaction capabilities — 3,000+ smart hardware categories, covering 200+ countries and regions, enabling AI Agents to control everything out of the box"

根据 Skill Creator 最佳实践：

"Include both what the Skill does and specific triggers/contexts for when to use it."
"The description is the primary triggering mechanism for your skill."

改进建议

description: >
  Control Tuya smart home devices via natural language. Use when the user asks to:
  (1) control smart devices — turn on/off lights, AC, plugs, adjust brightness/temperature/mode,
  (2) query device status or list devices,
  (3) manage homes and rooms,
  (4) check weather by location,
  (5) send notifications — SMS, voice call, email, or App push,
  (6) view device data statistics (e.g. energy consumption).
  Supports 3,000+ device categories across 200+ countries.
  Requires TUYA_API_KEY environment variable.

关键原则：把"什么场景触发"放在最前面，营销语言移到最后或删除。

三、工作流设计 — 缺少关键场景

现有优点

  • 5 个核心 Workflow 步骤清晰
  • Workflow 1（Device Control）的设备匹配分支逻辑写得很好

缺失的工作流

缺失场景影响
设备状态查询（独立）用户说"客厅灯是开着的吗？"没有对应工作流
多设备批量控制"关掉所有灯"、"把全屋灯调暗"无引导
设备离线处理设备 online: false 时应提前告知用户，而非尝试控制后失败
自然语言映射指南"调暗灯光" → 读取当前 bright_value → 减少 N，而非设为固定值

改进建议

增加 Workflow 6：设备状态查询

### Workflow 6: Device Status Query

When the user asks "Is the living room light on?" or "What's the AC set to?":

1. Locate the device (Workflow 1 Step 1)
2. Call "Get Single Device Detail" API
3. Check the `online` field — if false, inform the user the device is offline
4. Read `properties` values and translate to natural language

增加 Workflow 7：多设备批量控制

### Workflow 7: Multi-Device Control

When the user says "Turn off all lights" or "Set all ACs to 26 degrees":

1. Call "List All Devices" API
2. Filter by category or keyword matching
3. For each matching device, execute Workflow 1 Steps 3-5
4. Aggregate results and report successes/failures

在 Workflow 1 Step 2 后增加离线检查

2.5. **Check online status** — If `online` is false, inform the user:
     "Device XX is currently offline. Please check its power and network connection."
     Do not proceed to issue commands.

四、Python SDK (tuya_api.py) — 健壮性不足

问题清单

问题严重程度说明
无请求超时高requests.get/post 无 timeout 参数，可能无限阻塞
无响应校验高不检查 success 字段，Agent 需自行判断
CLI 无参数校验中args[0] 直接取值，缺少参数时抛 IndexError
无重试机制中网络抖动直接失败
无 HTTP 状态码检查中不调用 resp.raise_for_status()
无返回类型提示低返回值类型不明确

改进建议

添加 timeout + retry + 响应校验

import requests
from requests.adapters import HTTPAdapter
from urllib3.util.retry import Retry

class TuyaAPIError(Exception):
    def __init__(self, code, msg):
        self.code = code
        self.msg = msg
        super().__init__(f"Tuya API error {code}: {msg}")

class TuyaAPI:
    def __init__(self, api_key=None, base_url=None, timeout=30):
        # ...existing init...
        self.timeout = timeout
        retry = Retry(total=3, backoff_factor=0.5,
                      status_forcelist=[500, 502, 503, 504])
        self.session.mount("https://", HTTPAdapter(max_retries=retry))

    def _get(self, path: str, params: dict = None) -> dict:
        url = f"{self.base_url}{path}"
        resp = self.session.get(url, params=params, timeout=self.timeout)
        resp.raise_for_status()
        data = resp.json()
        if not data.get("success"):
            raise TuyaAPIError(data.get("code"), data.get("msg"))
        return data

七、安全性

当前优点

  • API Key 通过环境变量传递（不硬编码）
  • 明确列出了不支持的安全敏感操作（门锁、摄像头等）

缺失的安全指导

缺失项建议
TLS 验证确认 requests 默认验证 SSL，代码中不应出现 verify=False
Rate Limiting告知 Agent 涂鸦 API 有频率限制，连续控制多设备时需间隔
API Key 泄露提醒不要将 API Key 输出到日志或返回给用户
properties 输入验证控制命令的属性值应校验 typeSpec 范围，避免发送越界值

改进建议

在 SKILL.md Important Notes 中补充：

8. Never log or display the TUYA_API_KEY value in output
9. Validate property values against the Thing Model typeSpec before issuing commands
   (e.g., brightness must be within min-max range)
10. When controlling multiple devices in sequence, add a brief delay between requests
    to avoid rate limiting

八、其他改进建议

8.1 使用方式优先级不清晰

SKILL.md 列出了 3 种使用方式（Python SDK / CLI / curl），但没有明确告诉 Agent 优先用哪种。

## Usage

**Always prefer Method 1 (Python SDK)** — it handles authentication, URL resolution,
and JSON serialization automatically. Use CLI or curl only for debugging.

8.2 天气查询的坐标推断缺失

Workflow 4 说"获取用户经纬度"，但没提到 Home API 返回的 latitude / longitude 可以直接使用。

1. Obtain coordinates:
   - First check if the user's home has location set
     (from home list API — latitude/longitude fields)
   - If not, ask the user for their city and convert to coordinates

8.3 Thing Model 的二次解析提醒不足

result.model 是 JSON 字符串需二次解析，这是一个容易出错的点。当前在 references/device-control.md 中有提及，但 SKILL.md 的 Workflow 中没有强调。

3. **Query capabilities** — Call "Query Device Thing Model" API.
   **Important**: The `result.model` field is a JSON **string** that must be parsed
   again (e.g., `json.loads(result["model"])`) to obtain the property definitions.

8.4 缺少版本化策略

  • 没有提到 API 版本变更时的应对方案
  • 建议在 SKILL.md 底部注明当前 API 版本和最后验证日期

8.5 缺少多语言/本地化指导

  • 设备名称可能是中文、英文或其他语言
  • 模糊匹配时需考虑多语言场景
  • 建议在 Workflow 1 Step 1 补充提醒
